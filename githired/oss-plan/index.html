Open-source · call-to-action plan

Open-source the tracker. Keep the AI hosted.

A six-week sequenced plan to publish GitApplied as MIT open-core — the Kanban + auth + billing scaffold goes public, the AI services and scrapers stay private, the hosted SaaS at app.gitapplied.com keeps shipping unchanged.

Model: open-core License: MIT (public repo) Contributor terms: DCO sign-off Drafted: 2026-05-27
§ 1 · Context

Why now, and what's at stake.

Open-sourcing buys distribution, hiring signal, community goodwill, and the kind of credibility that makes a solo SaaS look serious. But done naïvely it also hands competitors (Teal, Huntr, Simplify) a free clone and lets self-hosters bypass the paid tiers. The constraint is to publish enough that "open-source" is an honest claim, while keeping the ~30–40% of code that constitutes the moat — prompts, scrapers, AI services — out of public view.

The codebase already has a clean natural seam: internal/auth/features.go tier-gates Free vs Base vs Premium, and that gate maps almost 1:1 to the open-core line. Free-tier code (Kanban, manual CRUD, basic URL scrape, auth, billing scaffold) goes public; paid-tier code (resume tailoring, cover letter generation, skill matching, structured posting breakdown) stays private. No invented architecture, no refactor war — just promotion of an existing boundary into a repository boundary.

Two non-negotiables: (1) the hosted SaaS keeps shipping unchanged — paid users notice nothing, (2) nothing currently sensitive (live keys, AWS account IDs, the MARKETING.md playbook, personal identity) ends up in the public repo or its history.

§ 2 · Strategic posture

Two repos, one product.

The public repo is a working job tracker on its own. The private repo turns it into the hosted SaaS by binding real AI providers and scrapers at startup. Same Go module imports, two different main.go entrypoints.

Public
github.com/gitapplied/gitapplied
MIT
  • Kanban board UI, job CRUD, notes, contacts
  • Auth (JWT, email verification, password reset)
  • Billing scaffold (Stripe wiring, tier middleware)
  • Basic URL scrape (HTML meta-tag fetch)
  • ai.Provider interface + no-op default
  • scrape.Scraper interface + generic default
  • Frontend shell, Dockerfile, example Terraform
Private
github.com/gitapplied/gitapplied-core
All rights reserved
  • internal/prompts/ — tuned LLM prompts
  • Resume tailoring + cover letter + suggestion services
  • Skill matching + scoring
  • Site-specific scrapers (LinkedIn, Indeed, Greenhouse, Lever, Workday)
  • Browser extension source
  • MARKETING.md, PRD.md, production infra docs
  • SaaS entrypoint that wires real providers

Trademark, not just license. MIT lets anyone use the code; a TRADEMARK.md file reserves the "GitApplied" name and logo. Forks can exist, but they can't ship as GitApplied. DCO over CLA — sign-off in commits is enough to keep the option open to dual-license later without forcing contributors through a legal agreement.

§ 4 · The technical seam

Two interfaces. The split lives here.

Introduce these in the existing monorepo first (during the private-fork refactor phase). Once handlers depend on the interfaces instead of concrete services, the AI implementation can move out cleanly. The hosted SaaS keeps importing the real implementation; the public binary binds the no-op.

// internal/ai/provider.go (PUBLIC) type Provider interface { EnrichJobPosting(ctx context.Context, rawHTML string) (Job, error) TailorResume(ctx context.Context, resume Resume, job Job) (Resume, error) GenerateCoverLetter(ctx context.Context, resume Resume, job Job, sample string) (string, error) SuggestResumeEdits(ctx context.Context, resume Resume, job Job) ([]Suggestion, error) MatchSkills(ctx context.Context, resume Resume, job Job) (MatchResult, error) PolishApplicationAnswer(ctx context.Context, prompt, draft string) (string, error) } // internal/ai/noop.go (PUBLIC) — default; returns ErrAINotConfigured. // Handlers detect this and respond HTTP 501 with a helpful body // pointing self-hosters at app.gitapplied.com or to BYO provider.
// internal/scrape/scraper.go (PUBLIC) type Scraper interface { Fetch(ctx context.Context, url string) (RawPosting, error) } // Public default: generic HTML + meta-tag scrape (current ExtractBasicFromURL). // Private impl: per-site adapters wired in at server start.

Wiring. cmd/server/server.go accepts Provider and Scraper as constructor args. The public main.go binds no-op + basic. The private SaaS entrypoint in gitapplied-core/cmd/saas imports the public module and binds real implementations.

Frontend impact: none required immediately. The UI already gates premium features by tier. When AI endpoints return 501 in self-hosted mode, surface a friendly "AI features require gitapplied.com hosted backend or a custom provider" banner using the existing tier-gate component.

§ 5 · Pre-launch checklist

Gating items. In order. Private fork only.

From the secrets audit: a .env with live OpenAI/AWS/Stripe/JWT keys was committed at some point. Assume compromised. Rotate first, then everything else.

  1. 1
    Rotate every credential that was ever in git history. Critical
    OpenAI, AWS access keys, Stripe secret + webhook secret, JWT secret, Postgres password. Do this regardless of the rest of the plan.
  2. 2
    Use a squashed initial commit. Do not try to scrub existing history.
    Create the new public repo from a curated tree as a single initial commit. Loses blame/git-log, but eliminates the entire class of "did we miss a file in an old commit" risk. The existing private repo stays as-is for internal continuity.
  3. 3
    Scrub identifiers from anything that goes public.
    AWS account ID 728821956014 → <ACCOUNT_ID> · real instance/subnet/bucket IDs → placeholders · gabriel-admin IAM references → admin · "Gabriel Aboy" / personal-email references → "Maintainer" or remove. The domain app.gitapplied.com is fine — it's already public.
  4. 4
    Add LICENSE (MIT) at root.
    Standard MIT text with current year + "GitApplied Maintainers" as copyright holder.
  5. 5
    Add SECURITY.md with a dedicated disclosure email.
    Use security@gitapplied.com, not a personal address. State response SLA (e.g. 72 hours) and supported versions.
  6. 6
    Add .env.example with every required variable name and no values.
    Include comments noting which are required vs optional, and which gate which features.
  7. 7
    Add CONTRIBUTING.md noting DCO sign-off requirement.
    Brief: how to build, how to test, code style, "all commits must be Signed-off-by", triage policy.
  8. 8
    Add TRADEMARK.md reserving the GitApplied name and logo.
    Even though the code is MIT, the brand is not licensed. Forks may not call themselves GitApplied or use the logo.
  9. 9
    Verify CI passes with the no-op AI provider bound.
    make test must be green without any OpenAI key. This is the contract: self-hosters get a working tracker out of the box.
  10. 10
    Run gitleaks + trufflehog on the curated tree. Critical
    Belt-and-suspenders secret scan. Zero findings required before pushing.
§ 6 · Repo creation strategy

Extract first, refactor in place, publish last.

The order matters: move IP-sensitive files to the new private repo before introducing the interface seam, so the diff stays narrow. Then verify the hosted SaaS still works end-to-end before any public push.

Today (private monorepo: GitHired) │ ├─→ extract IP-sensitive files to new private repo gitapplied-core │ (git filter-repo, preserve history of moved files) │ ├─→ refactor in place: add ai.Provider + scrape.Scraper interfaces, │ swap call sites to depend on interfaces instead of concrete services │ ├─→ verify hosted SaaS still works (deploy to staging.gitapplied.com) │ └─→ publish curated tree as new repo gitapplied (squashed initial commit, MIT, CI, branch protection, issue templates)

Hosted production then becomes: gitapplied-core imports github.com/gitapplied/gitapplied as a Go module, binds real providers, builds the SaaS binary. The public binary remains usable standalone — it just returns 501 for AI features.

§ 7 · Launch sequence

Quiet first. Loud second. Never claim "fully open source."

Messaging guardrail: say "open-core: the tracker is MIT, the AI is hosted." Misrepresentation gets punished hard on HN — honesty is the marketing.

Stage 1
Quiet pre-launch
  • Invite ~10 trusted founders / devs to clone and self-host for a week
  • Collect friction list — what breaks, what's confusing
  • Fix top 5 issues before any wider push
Stage 2
Soft launch
  • Publish the public repo
  • Post in a single low-key channel (existing audience or one subreddit you already participate in)
  • No press, no Show HN yet
Stage 3
Loud launch
  • Show HN + Indie Hackers + r/SaaS
  • Blog post on gitapplied.com: why open-source, what's private and why
  • Link to hosted product as the easy path
Stage 4
Discoverability
  • PR to awesome-selfhosted
  • GitHub topic tags: job-search, resume, kanban, self-hosted
  • One-screen demo gif at the top of README
§ 8 · Sustainment

Plan the steady-state before launch day.

  • Support load. Budget 2–4 hours per week initially for issues + PR triage. If it overruns, throttle by closing stale issues aggressively or disabling Discussions.
  • Governance. Sole maintainer. Document that decisions are yours; community contributions welcome but not promised to merge.
  • DCO bot. Install the DCO GitHub App so PRs without Signed-off-by are blocked automatically — no manual policing.
  • Roadmap visibility. Post a quarterly "what's coming in the public repo" issue. Keep paid-only features off the public roadmap entirely.
  • Release cadence. Tag the public repo monthly. The private SaaS can deploy daily as it does today.
§ 9 · Risks & guardrails

What can go wrong, and what blunts it.

RiskMitigation
Competitor lifts the public tracker and rebrands as their own. Trademark on the GitApplied name + logo. Their fork can exist but can't trade on your brand. Speed-of-iteration is your remaining moat.
Self-hoster guesses the prompts and recreates the AI features. Acceptable. The prompts are not the whole moat — scrapers + ongoing tuning + cost amortization are. Track this; if a quality clone emerges, reassess.
Old git history leaks if we scrub-and-republish the existing repo. Avoided by using a fresh squashed initial commit. No history = no leakable past.
Free-tier hosted users churn to self-hosting. Likely net-neutral: self-hosters were never going to pay anyway. Some convert later when they hit operational friction (OpenAI bills, scraper rot). Watch the cohort.
Maintainer burnout from issue load. Strict triage policy in CONTRIBUTING.md: "Issues may be closed without response if low-effort or out-of-scope." Disable Discussions if needed.
You decide later you want to relicense (e.g. to BSL or AGPL). DCO + MIT lets you ship new versions under any license. Old versions stay MIT forever — that's the deal you make with contributors up-front.

  2. 2
    Hosted SaaS unchanged on staging.
    Deploy the new architecture to staging.gitapplied.com from gitapplied-core (imports public module + binds real providers). Walk through every paid-tier flow: scrape a LinkedIn job, generate tailored resume, generate cover letter, get skill match. Identical to prod today.
  3. 3
    Secret scan returns zero findings.
    gitleaks detect --source . --no-git and trufflehog filesystem . both clean on the curated tree.
  4. 4
    No personal info in the public tree.
    rg -i 'gabriel|aboy.gabriel|728821956014' returns nothing.
  5. 5
    License compliance clean.
    go-licenses report ./... shows no GPL/AGPL deps. license-checker on web/ shows the same.

    Frontend degrades gracefully.
